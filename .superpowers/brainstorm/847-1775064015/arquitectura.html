Arquitectura: 3 Enfoques de Refactorización

¿Cómo transformamos el marketplace en tienda única?

A

Fusión Total — 2 roles: admin + buyer

Eliminar completamente seller. El admin hereda TODAS las funciones del seller. Una sola tabla company_config reemplaza stores. Rutas: /admin/* para todo el panel, /cuenta/* para compradores.

Pros

  • Máxima simpleza
  • Sin ambigüedad de roles
  • Schema limpio

Cons

  • Más cambios en DB
  • Hay que migrar lógica seller → admin

⭐ RECOMENDADO

B

Renombrado Suave — seller se convierte en admin

Mantener la tabla stores pero con un solo registro (la empresa). El rol seller pasa a llamarse admin en el enum. El dashboard /dashboard se convierte en panel admin. Cambios mínimos.

Pros

  • Cambios mínimos en código
  • Menos riesgo de romper funciones

Cons

  • Queda código de marketplace disfrazado
  • Confuso para mantenimiento futuro
C

Híbrido — Eliminar stores, fusionar panels

Eliminar stores y store_id del schema. Fusionar (seller)/dashboard y (admin)/admin en un solo (admin)/panel. El admin ve productos, pedidos, clientes, inventario, promociones en un solo panel unificado.

Pros

  • Panel admin más potente
  • Schema verdaderamente limpio
  • Base sólida para escalar

Cons

  • Trabajo medio — más que B, menos que rebuild
